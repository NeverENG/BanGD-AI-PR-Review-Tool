# Few-shot 范例：复用 protobuf 字段号 → 兼容契约

维度：接口与兼容契约。展示"破坏线上/磁盘格式契约"类问题。

## 输入（PR 改动节选）
为"清理"协议，删了一个旧字段并把字段号 3 复用给了新字段：
```proto
message GetResp {
  bytes value = 1;
  // string old_owner = 3;  // 删除
  int64 version = 3;        // 复用 3 号
}
```

## 通用评审者会给的解法
> 改了字段，记得同步更新客户端代码。

## BanGD 的理想评审
- **位置**：`test_grpc/kv.proto` `GetResp`
- **严重度**：阻塞
- **类型**：兼容
- **问题根因**：**复用了已用过的字段号 3**。protobuf 用字段号做线上身份：滚动升级或旧客户端发来的老消息里，3 号承载的是 `old_owner`（string），新代码却按 `version`（int64）解析——**静默的数据错乱**，不是解析报错，更难发现。根因是把 proto 当普通结构体改，忽视了它是跨版本的兼容契约。
- **为什么低级解法不够**：同步客户端只覆盖"同时升级"的理想情况；分布式系统里新旧版本必然共存一段时间，老消息、持久化的老数据、回放的老日志都会撞上这个被复用的字段号。
- **架构级方案**：遵守**只增不改**的演进纪律：新字段用**新的字段号**；废弃字段用 `reserved` 标记字段号与名字，永不复用。需要语义迁移时通过**版本协商**或在读路径**同时兼容新旧两种格式**过渡。磁盘/WAL 记录格式同理——带版本号、保留旧格式读取路径。
- **代价/收益**：代价是字段号只增不减、proto 看起来"不够整洁"；收益是滚动升级与历史数据永远可正确解析，这是任何对外格式不可妥协的前提。
