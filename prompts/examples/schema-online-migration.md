# Few-shot 范例：阻塞式 DDL → 在线迁移

维度：Schema 演进与迁移。展示"大表迁移阻塞生产"类问题。

## 输入（PR 改动节选）
迁移逻辑对全表加写锁后原地改写每一行：
```go
func migrateAddColumn(t *Table) error {
	t.Lock(); defer t.Unlock()
	for _, row := range t.AllRows() { row.rewriteWithNewColumn() }
	return nil
}
```

## 通用评审者会给的解法
> 把整个迁移包进一个事务，保证原子性。

## BanGD 的理想评审
- **位置**：`migrate add column`
- **严重度**：阻塞
- **类型**：schema
- **问题根因**：迁移在持有**全表写锁**期间遍历改写所有行，大表上会把所有读写阻塞数分钟到数小时。根因是把"一次性原地改写"当成了迁移模型，没有考虑在线性。
- **为什么低级解法不够**：包进事务只会让锁持有更久、回滚成本更高，生产阻塞问题反而更严重——它解决的是"原子性"，而真正的问题是"可用性"。
- **架构级方案**：采用**影子表 + 双写 + 后台回填 + 原子切换**的在线迁移：建带新列的影子表；迁移期间对两表双写；后台分批回填存量数据（每批短临界区）；校验一致后原子地把读路径切到影子表。全程不阻塞前台读写。
- **代价/收益**：代价是迁移逻辑显著复杂、迁移期间双写有额外开销与磁盘占用；收益是大表迁移对线上零（或秒级）阻塞，这是生产数据库不可让步的可用性底线。
