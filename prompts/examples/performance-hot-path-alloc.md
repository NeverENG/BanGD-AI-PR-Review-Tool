# Few-shot 范例：热路径上的每行分配 → 移出热路径

维度：热路径性能。展示"在最高频路径引入开销"类问题。

## 输入（PR 改动节选）
为方便排查，在点查热路径里加了结构化日志：
```go
func (m *MemTable) Get(key []byte) ([]byte, error) {
	slog.Debug("get", "key", fmt.Sprintf("%x", key)) // 每次 Get 都执行
	...
}
```

## 通用评审者会给的解法
> 把日志级别调成 Debug 就行（默认不输出）。

## BanGD 的理想评审
- **位置**：`storage/zstorage/memtable.go` `Get`
- **严重度**：重要
- **类型**：性能
- **问题根因**：`Get` 是 QPS 最高的点查热路径，`fmt.Sprintf("%x", key)` **无论日志是否输出都会执行**——每次调用一次堆分配 + 格式化，在高并发下显著抬升 GC 压力与延迟。根因是把可观测性代码直接塞进了最热的路径。
- **为什么低级解法不够**：调成 Debug 级别只是不**输出**，但 `Sprintf` 的实参在调用 `slog.Debug` 前就已求值——分配照样发生。这是对 Go 求值语义的误解，问题并未消除。
- **架构级方案**：把开销移出热路径——用**惰性求值**（`slog` 的 `LogValuer` 或先 `Enabled()` 判断再格式化），或改为**采样/计数**式观测（原子计数 + 周期性导出），而不是逐次格式化。原则：热路径只做点查本身该做的事，可观测性必须零额外分配或可关断。
- **代价/收益**：代价是日志写法稍复杂；收益是热路径恢复零额外分配，吞吐与延迟不受观测代码拖累。
