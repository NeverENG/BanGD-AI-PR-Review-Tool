## 方法

- **语料**：6 个**真实** BanDB PR（`eval/cases.json`，由 `eval/build-corpus.mjs` 从 GitHub 冻结）。标签来源：作者明示意图或**维护者在后续修复 PR 中确认**的缺陷——例如 #58 的崩溃一致性与 Stat→Seek 竞态，由维护者在修复 PR #63 中确认并修复。
- **防泄漏**：真实 PR 描述常直接点明缺陷（如 #43 自述「刻意用并发写法检验 BanGD」），会把答案喂给模型。故语料中**丢弃真实描述、改用中性事实化的一句话**，只有 diff（代码）进入模型。
- **匹配规则**：预测 finding 与期望 finding 按「文件名 + 类型集合」做一对一贪心匹配（见 `src/eval/score.ts`），只认结构锚点、不比对措辞，避免被表述方式钻空子。
- **单变量 A/B**：两次运行只差一个变量——**对抗式复核关 / 开（0 票 vs 3 票）**；模型、提示词（含 few-shot）、上下文全部一致，每个配置用全新客户端以隔离 token 计数。
- **上下文**：仅 diff（`readFile` 返回 null），两配置一致，保证 A/B 可比。

## 结论（解读由数据手写；上方表格为机器生成）

**关键发现：在 DeepSeek（廉价兼容模型）上，对抗式复核适得其反——它把真 finding 连同误报一起杀掉了。**

- **召回率 100% → 0%**：baseline 找全了全部 3 条真 finding（#43 并发自增、#58 的崩溃一致性 + Stat→Seek 竞态）；对抗式复核的反驳者（同样是 DeepSeek）把这 3 条**全部误杀**，复核后两案归零。
- **精确率 21.4% → 0%（不是提升！）**：架构误报数确实从 11 条降到 1 条，但因为分子（真 finding）被清零、唯一存活的还是个误报（#35），精确率反而**掉到 0**。F1 35.3% → 0%。把"误报数下降"误读成"精确率提升"，正是这条数据要破除的错觉。
- **成本 +173%**（74,626 → 203,878 token）：多烧 2.7 倍 token，换来的是更差的准确率。

**为什么**：对抗式复核把判断力全押在反驳者模型上（"严格多数否决 + 证据不足即判误报"）。DeepSeek 这类弱模型反驳时倾向"看着证据不足就否"，于是把需要数据库架构推理才能坐实的真隐患也一并否掉。**复核的质量上限 = 反驳者模型的质量上限。**

**这恰好用数据印证了 DESIGN 的两个判断**：
1. **模型选择是分水岭**（§二）：DeepSeek 只配冒烟"链路通不通"。baseline 召回满分却满屏误报、复核又矫枉过正——都是弱模型推理力不足的两种表现。真实准确率必须在 **Opus（或同档）重跑**，本报告绝对数字只代表"廉价端点"路径，不可外推到生产。
2. **同质反驳者是缺陷**（§六.3）：N 个同质 DeepSeek 反驳者只会齐声否决。这直接指明下一步——做**视角分化的反驳者**（各自从不同失败维度切入），而非简单加票数。

**行动建议**：
- `verify_votes` 默认值应**随模型强度调整**；弱模型上 3 票/条是净负，应调低或关闭。
- 在 Opus 档重跑本评测，看复核能否在**不杀真 finding** 的前提下削减误报。
- 优先实现 §六.3 的视角分化反驳者，再用本语料 A/B 验证。
- baseline 在两处**正确修复**（#40/#35）上无中生有报了 3–4 条架构 finding——说明弱模型在并发类 diff 上"易被诱导"，这也是复核本想解决、却用力过猛的问题。
