# Few-shot 范例：缓冲池切片逃逸 → 所有权边界

维度：内存管理与生命周期。展示"复用内存的所有权边界"类问题。

## 输入（PR 改动节选）
`SSTable.Get` 为减少分配，直接返回指向 `sync.Pool` 读缓冲的切片：
```go
buf := s.pool.Get().([]byte)
n, _ := s.f.ReadAt(buf, off)
val := buf[hdr:n]      // 指向 pool 缓冲的切片
s.pool.Put(buf)        // 缓冲已归还
return val, nil        // 但 val 仍指向它
```

## 通用评审者会给的解法
> 在 `Put` 之前 `copy` 一份再返回。

## BanGD 的理想评审
- **位置**：`storage/zstorage/SSTable.go` `Get`
- **严重度**：阻塞
- **类型**：内存
- **问题根因**：返回值 `val` 与被归还的 pool 缓冲共享底层数组——`Put` 后该缓冲可能被另一个 goroutine 取走并覆写，调用方读到的是被污染的数据（use-after-free 的 GC 语言版本）。根因是**所有权边界没划清**：池化内存的生命周期止于本函数，却让引用逃逸了出去。
- **为什么低级解法不够**：在这一处 `copy` 能堵住本例，但没有确立"池化缓冲不得逃逸"的规则——下一个读路径还会再犯，且每处都 `copy` 会抵消池化省分配的初衷。
- **架构级方案**：在**模块边界**统一所有权：读缓冲只在 SSTable 内部使用，`Get` 返回**调用方拥有**的内存（解码进入调用方传入的 buffer，或返回明确 own 的 []byte）；池化只在"取-用-还"都在同一函数内闭合时使用。用类型/接口让"借用 vs 拥有"在签名上可见。
- **代价/收益**：代价是边界处一次可控的拷贝或一个传入 buffer 的 API 调整；收益是从设计上根除整类 use-after-free，而非逐点打补丁。
