# Few-shot 范例：不一致的加锁顺序 → 死锁

维度：锁与隔离级别。展示"锁顺序"类问题。

## 输入（PR 改动节选）
新增的 compaction 路径按 `table → wal` 顺序加锁：
```go
func (m *MemTable) compact() {
	m.tableMu.Lock()
	m.walMu.Lock()
	...
}
```
而既有的 flush 路径是 `wal → table` 顺序加锁。

## 通用评审者会给的解法
> 给 `Lock` 加超时 / 用 `TryLock` 重试。

## BanGD 的理想评审
- **位置**：`storage/zstorage/memtable.go` `compact`
- **严重度**：阻塞
- **类型**：锁
- **问题根因**：compaction 以 `table→wal`、flush 以 `wal→table` 获取同两把锁，构成经典的**循环等待**——两者并发时必然死锁。根因是全局缺少**统一的锁获取顺序**约定。
- **为什么低级解法不够**：`TryLock`+重试把确定性死锁换成了活锁与无界重试延迟，且掩盖了"锁层级未定义"这一设计缺陷；超时则把死锁变成偶发超时错误，更难排查。
- **架构级方案**：确立并文档化**全局锁层级**（如恒为 `wal < table`），所有路径一律按该顺序获取；用一个封装好的"按序加锁"helper 强制执行，从源头杜绝逆序。更进一步可评估这两把锁能否合并为单一所有者，消除多锁协调。
- **代价/收益**：代价是梳理现有加锁点并统一顺序；收益是死锁在设计层面不可能发生，而非靠超时概率性规避。
