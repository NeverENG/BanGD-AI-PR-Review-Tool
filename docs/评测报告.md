## 方法

- **语料**：6 个**真实** BanDB PR（`eval/cases.json`，由 `eval/build-corpus.mjs` 从 GitHub 冻结）。标签来源：作者明示意图或**维护者在后续修复 PR 中确认**的缺陷——例如 #58 的崩溃一致性与 Stat→Seek 竞态，由维护者在修复 PR #63 中确认并修复。
- **防泄漏**：真实 PR 描述常直接点明缺陷（如 #43 自述「刻意用并发写法检验 BanGD」），会把答案喂给模型。故语料中**丢弃真实描述、改用中性事实化的一句话**，只有 diff（代码）进入模型。
- **匹配规则**：预测 finding 与期望 finding 按「文件名 + 类型集合」做一对一贪心匹配（见 `src/eval/score.ts`），只认结构锚点、不比对措辞，避免被表述方式钻空子。
- **单变量 A/B**：两次运行只差一个变量——**对抗式复核关 / 开（0 票 vs 3 票）**；模型、提示词（含 few-shot）、上下文全部一致，每个配置用全新客户端以隔离 token 计数。
- **上下文**：仅 diff（`readFile` 返回 null），两配置一致，保证 A/B 可比。

## 结论与诚实解读（手写）

**主结论（已证实）：反驳过度（over-refutation）已修复。**
上一版复核器（同质反驳者 + "存疑即否" + 多数否决）把真 finding 全杀了——召回 100%→0%（见文末"修复前后对比"）。本版**召回保持 33.3% 未降**：#43 的真并发 finding 在复核后**存活**（该案 R=100%），旧版则会误杀它。修复手段：A 保留默认、B 视角分化（整体判定）、C 全票才删。这个核心目标已达成，且**不受下列混淆因素影响**。

**次结论（方向性，被混淆，勿过度解读）：**
F1 22.2%→40%、精确率 16.7%→50%、架构误报 5→1 看似"复核在降误报"，但有三处混淆，必须点明：
1. **#35 的"改善"其实是解析错误**：优化配置下 #35 模型未返回可解析输出（降级为 0 finding），**不是复核删掉了它的 3 条误报**——这条精确率提升是假象。
2. **A/B 混淆了"复核效果"与"DeepSeek 生成随机性"**：baseline 与优化是**两次独立生成**，#40 优化配置最终 0 finding，无法区分是复核删的、还是这次生成本就少报。
3. #43 优化配置反而**多出一条 `性能` 误报**（P 100%→50%），说明复核**并未**清掉所有误报。

因此"复核降误报"在本数据上**只有弱支持**（主要来自 #40，且被混淆）。要得到干净的精确率结论，必须把评测改造为**"只生成一次 finding，再对同一批做 复核/不复核"**（见局限），消除生成随机性——这是下一步。

**新发现：DeepSeek 结构化输出不稳。** 12 次评审里 **3 次解析失败**（#58 两次、#35 一次，约 25%）。其中 #58 是召回锚点，**两次都失败 → 布隆案从未被评分**，召回上限被钉在 #43 一条上，方差极大。这再次印证 DESIGN §二：DeepSeek 只配冒烟链路，真实质量评测须用 Opus。

## 局限

- 语料仅 6 例（召回锚点 2 例 / 3 条 finding），数字方差大，结论是趋势性的而非统计显著。
- **A/B 不够干净（待修）**：当前两个配置各自**重新生成**一遍 finding，把"复核效果"与 DeepSeek 的生成随机性、解析失败混在一起。下一步应改为**"生成一次 finding → 对同一批做 复核/不复核"**，才能干净地归因复核对精确率的作用。
- **DeepSeek 解析失败率高**（本次约 25%，#58 两次、#35 一次），且恰好打掉了召回锚点 #58 → 召回实际只由 #43 决定。这是廉价模型的鲁棒性问题，非复核器所致。
- **全票否决（C）在强模型上可能过保守**：三个强反驳者里一个弃权就保留，可能放过本该删的误报。Opus 重跑时应**重新标定删除阈值**（如改 2/3），而非照搬全票。
- 标签虽锚定真实 PR 与维护者确认，仍含评审判断；匹配按文件+类型，可能与人工判定有出入。
- 仅用 diff 上下文（未喂改动文件全文/周边代码），低估了 BanGD 在完整上下文下的能力。
- 评测模型/端点见报告头部；用便宜的兼容端点冒烟与用 Opus 生产，质量不可直接外推。
