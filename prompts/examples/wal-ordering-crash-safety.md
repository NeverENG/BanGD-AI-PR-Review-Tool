# Few-shot 范例：写入顺序 → 崩溃恢复安全

第二个样板，刻意选在不同维度（存储/WAL），避免评审风格过拟合到"凡事都提双表"。同样展示根因→拒绝低级解法→架构方案→权衡的深度。架构级方案不一定是双表——这里的正解是恢复写入顺序的不变量。

---

## 输入（PR 改动节选）

PR 在 `storage/page.go` 的刷盘逻辑里，为了"减少一次 fsync 提升写入吞吐"，调整了顺序：先把脏页写入数据文件，再追加 WAL 记录。

```go
func (s *Store) Flush(p *Page) error {
    if err := s.dataFile.WriteAt(p.Bytes(), p.Offset()); err != nil { // 先写数据
        return err
    }
    return s.wal.Append(p.Record()) // 后写日志
}
```

---

## 通用评审者（Copilot）会给的解法

> 看起来能正常工作，写入吞吐确实提升了。也许补一句注释提醒顺序。
（通用评审者往往看不出这是崩溃安全问题，因为正常路径下结果正确。）

---

## BanGD 的理想评审

- **位置**：`storage/page.go` `Store.Flush`
- **严重度**：阻塞
- **类型**：存储 / 崩溃恢复
- **问题根因**：改动**违反了 WAL（Write-Ahead Logging）的核心不变量**——日志必须先于它所描述的数据落盘。现在先写数据后写日志，如果在两步之间崩溃，数据文件已被修改但 WAL 没有对应记录，**恢复时无法回滚也无法重做，数据库进入不可恢复的不一致状态**。正常路径"看起来正确"恰恰是它最危险的地方。
- **为什么低级解法不够**：
  - 加注释完全没解决问题，只是把炸弹留给下一个人。
  - "为省一次 fsync 而调换顺序"这个优化目标本身是合理的，但实现方式错了——它用**牺牲崩溃安全**换性能，这是数据库绝不能接受的交换。
- **架构级方案**：恢复 WAL 顺序不变量，并用**组提交 (group commit)** 达成原本想要的吞吐目标：
  - 顺序改回：先 `wal.Append` 并确保日志持久化，再写数据文件。
  - 想省 fsync，不是靠调换顺序，而是把多个并发事务的 WAL 记录**批量攒到一起、合并成一次 fsync**（group commit），在不破坏顺序不变量的前提下摊薄 fsync 成本。
- **代价/收益**：
  - 代价：组提交需要引入提交队列与短暂的批处理等待，单事务延迟略增；实现比简单调换顺序复杂。
  - 收益：崩溃安全不变量得以保持，恢复永远一致；同时通过批量化把 fsync 次数从"每事务一次"降到"每批一次"，吞吐提升通常**优于**原始的危险改法。鱼和熊掌兼得。
