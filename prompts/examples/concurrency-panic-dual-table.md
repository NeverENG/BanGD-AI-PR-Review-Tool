# Few-shot 范例：并发 panic → 双表设计

这是 BanGD 理想评审的样板。它展示了"识别根因 → 拒绝低级解法 → 提出架构级方案 → 诚实权衡"的完整深度。新范例应保持同样的高度。

---

## 输入（PR 改动节选）

PR 在 `cache/block.go` 给内存块缓存新增了一个统计字段，多个查询 goroutine 会在读路径上更新它：

```go
type BlockCache struct {
    blocks map[uint64]*Block
    hits   uint64 // 新增：命中计数
}

func (c *BlockCache) Get(id uint64) *Block {
    b := c.blocks[id]
    if b != nil {
        c.hits++ // 新增：多个 goroutine 并发执行，竞争 + map 并发读也存在
    }
    return b
}
```

CI 偶发崩溃：`fatal error: concurrent map read and map write`，以及 `hits` 计数明显偏小。

---

## 通用评审者（Copilot）会给的解法

> 给 `Get` 加锁，或者把 `hits++` 换成 `atomic.AddUint64`。

---

## BanGD 的理想评审

- **位置**：`cache/block.go` `BlockCache.Get`
- **严重度**：阻塞
- **类型**：并发 / 内存
- **问题根因**：`BlockCache` 把**读路径**和**可变共享统计状态**放进了同一个结构，且 `blocks` map 本身在并发读路径上没有任何保护。崩溃的直接原因是 map 并发读写，但**根因是所有权模型错误**——一个本应是只读热路径的查询路径，被塞进了需要写同步的状态。
- **为什么低级解法不够**：
  - 给 `Get` 加互斥锁会把**整个点查热路径串行化**——这是数据库最高频的路径，吞吐会断崖式下跌，等于为了统计数字牺牲核心性能。
  - 仅把 `hits++` 改 `atomic` 能消除计数竞争，但**完全没解决 `blocks` map 的并发读写崩溃**——崩溃照旧。它只是让人误以为修好了。
  - 两种改法都停在"消除这一处竞争"，没有质问"读路径上为什么会有写"。
- **架构级方案**：**读写分离的双表/影子表设计**。
  - 把 `blocks` 拆成不可变的**只读快照**（查询路径只读它，零锁、零竞争）+ 一张承接更新的**写表**，后台以 copy-on-write 周期性地把写表合并发布为新快照（原子换指针）。
  - 统计计数从读路径剥离：用 **per-shard / per-goroutine 计数器**，读路径只对自己分片的本地计数 +1（无共享），需要总数时再聚合。彻底消除热路径上的跨 goroutine 同步。
- **代价/收益**：
  - 代价：实现复杂度上升（需快照发布机制 + 分片计数聚合）；快照有秒级可见性延迟；写表合并有额外内存峰值。
  - 收益：查询热路径**完全无锁无竞争**，并发崩溃从根上消除，吞吐随核数线性扩展，统计精确。对一个点查 QPS 是核心指标的存储引擎，这个权衡几乎总是值得。
