# BanGD 评审 Rubric（数据库内核）

这是 BanGD 评审 BanDB（Go 编写的数据库引擎）源码改动时的领域清单。它定义"作为资深数据库系统工程师，应该追问什么、该往哪个架构方向推"。

**使用原则**
- 不是逐条打勾，而是用它锚定**追问的深度**。看到症状要问到根因，看到根因要给架构级方案。
- 每个维度下列的是「触发信号」+「该追问的架构问题」。如果一条改动命中某个信号，就按对应问题展开。
- **永远拒绝"最低级解法"作为最终建议**：把 async 改 sync、无脑加锁、加 sleep、吞异常——这些可以提，但必须同时给出从架构层面消除问题的方案，并比较两者代价。
- 输出每条 finding 严格遵循：`问题根因 → 为什么低级解法不够 → 架构级方案 → 代价/收益`。

---

## 1. 并发与同步 (Concurrency & Synchronization)

**触发信号**：新增 goroutine、共享可变状态、`go func`、channel、被多线程访问的字段、map 并发读写、计数器、缓存。

**该追问的架构问题**
- 这个共享状态**本该被共享吗**？能否通过数据结构拆分（如分片、双表/影子表、per-shard 所有权）让它根本不需要跨 goroutine 同步？
- 如果必须同步，锁的**粒度和位置**对吗？是不是把粗锁加在了热路径上？能否用更细的所有权模型替代锁？
- 是否存在「并发 panic」类问题（并发写 map、slice 扩容竞争、关闭已关闭 channel）？根因是所有权不清，不是"少加了锁"。
- 读多写少时，是否该用 RCU / copy-on-write / 双缓冲，而不是读写锁？

## 2. 内存管理与生命周期 (Memory & Lifecycle)

**触发信号**：内存块/缓冲池、`unsafe`、手动复用对象、`sync.Pool`、对象在多个组件间传递、释放/归还逻辑。

**该追问的架构问题**
- 这块内存的**所有者是谁**，生命周期边界在哪？多个 goroutine 同时持有引用时，谁负责释放？
- 是否存在 use-after-free / double-free / 数据被提前覆盖的风险？这通常要靠**所有权架构**解决，不是加锁。
- 复用对象（pool）的归还时机是否可能在仍被读取时发生？

## 3. 锁与隔离级别 (Locking & Isolation)

**触发信号**：事务路径、`Lock/Unlock`、隔离级别、MVCC 版本链、可见性判断。

**该追问的架构问题**
- 锁顺序是否一致？是否引入了潜在死锁（多锁获取顺序不一致）？
- 改动是否破坏了隔离级别保证（脏读/不可重复读/幻读）？MVCC 版本可见性判断是否仍正确？
- 长事务/长持锁是否会阻塞热路径？能否缩短临界区或改为乐观并发？

## 4. 存储格式、WAL 与崩溃恢复 (Storage / WAL / Crash Safety)

**触发信号**：磁盘格式、序列化、WAL/redo/undo、fsync、checkpoint、文件头、版本号。

**该追问的架构问题**
- 改动是否改变了**磁盘格式或 WAL 记录格式**？是否向后兼容、旧数据能否读取、是否有版本号与迁移路径？
- 写入顺序对崩溃恢复安全吗？是否遵守 WAL 原则（先写日志再改数据）？fsync 时机对吗？
- 在写入中途崩溃，恢复后状态是否一致？是否引入了不可恢复的中间态？

## 5. Schema 演进与迁移 (Schema Evolution)

**触发信号**：表结构、列定义、元数据、`ALTER`、迁移脚本。

**该追问的架构问题**
- 是否需要在线迁移？大表上的阻塞性 DDL 会不会卡死生产？能否用影子表/双写过渡？
- 新旧 schema 共存期间，读写路径是否都能正确处理两种版本？

## 6. 热路径性能 (Hot-Path Performance)

**触发信号**：查询执行、点查/扫描、索引、序列化、每行/每条都会走的代码。

**该追问的架构问题**
- 改动是否在热路径上引入了额外分配、拷贝、锁竞争、syscall、或 O(n) 退化？
- 是否有**架构级的性能改善机会**（如双表/物化、批处理、零拷贝、预计算索引）被忽略了？BanGD 应主动提出这类机会，不只是不退化。

## 7. 资源与泄漏 (Resource Management)

**触发信号**：goroutine 启动、文件句柄、连接、`defer Close`、context 取消、后台循环。

**该追问的架构问题**
- 每个新 goroutine / 句柄 / 连接是否都有明确的退出/释放路径？关停时是否优雅退出？
- context 取消是否被正确传播？后台循环是否可能泄漏？

## 8. 错误处理与崩溃安全 (Error Handling)

**触发信号**：`panic`/`recover`、错误被吞、`_ =`、忽略返回值、partial write。

**该追问的架构问题**
- 错误是否被静默吞掉、掩盖了真正的不变量被破坏？
- panic 是不变量被破坏的信号——根因是什么？修复应消除不变量破坏，而不是 `recover` 兜住。

## 9. 接口与兼容契约 (API / Compatibility Contracts)

**触发信号**：导出符号、协议、RPC、客户端可见行为、配置项默认值。

**该追问的架构问题**
- 是否破坏了对外契约（线协议、API、配置语义）？是否需要版本协商或弃用周期？
