# Few-shot 范例：吞掉 fsync 错误 → 不变量必须显式维护

维度：错误处理与崩溃安全。展示"吞错掩盖不变量被破坏"类问题。

## 输入（PR 改动节选）
提交路径忽略了 WAL 落盘的返回错误：
```go
func (w *WAL) Commit(rec []byte) {
	w.append(rec)
	_ = w.f.Sync()   // 忽略 fsync 错误
}
```

## 通用评审者会给的解法
> 把错误记一条日志。

## BanGD 的理想评审
- **位置**：`Raft/raft_wal.go` `Commit`
- **严重度**：阻塞
- **类型**：错误处理
- **问题根因**：`fsync` 失败被 `_ =` 吞掉，但提交流程仍当作成功返回。这破坏了**持久性不变量**：上层以为数据已落盘并对客户端确认，实际可能丢失。根因是把"尽力而为"的态度用在了一个必须成立的不变量上。
- **为什么低级解法不够**：记日志只是事后留痕，调用方依然收到"提交成功"并继续推进状态机——不一致已经发生。日志不能替代"违反不变量时必须中止"。
- **架构级方案**：让 `Commit` **返回 error 并向上传播**，fsync 失败时**拒绝确认该提交**、触发既定的失败处理（重试/降级只读/上报）。在类型层面强制：凡是关乎持久性的调用，其错误不可被丢弃（让签名返回 error，并在 lint/审查中禁止对它 `_ =`）。
- **代价/收益**：代价是 `Commit` 及其调用链要处理错误路径；收益是持久性承诺要么成立、要么明确失败，绝不静默撒谎——这是数据库的底线。
